# 开发记录 / 任务总结

> 记录 MomentLog 从需求定义到首版可运行骨架搭建的全过程，以及依赖安装过程中遇到的问题与解决方案。
> 最近更新：2026-07-11

## 1. 需求演进（PRD v0.1 → v0.6）

PRD 经 6 次迭代，累计确认 12 项关键决策（第 10 节无遗留待确认项）：

| 版本 | 关键变更 |
|------|----------|
| v0.1 | 初始：事件 → 记录 两层模型 |
| v0.2 | 新增「主题(Topic)」为顶层容器，首页改为主题列表页；层级 主题 → 事件 → 记录 |
| v0.3 | 主题级分类标签：①记录增改可勾选标签（非必选）②每主题独立标签集 ③主题内按标签搜索（任一/全部匹配） |
| v0.4 | 架构转向「客户端+服务端、服务端存储、多端同步、多用户就绪」：MVP 单用户(默认 user_id=1)但服务端按多用户设计；图片上传服务端 uploads/；Naive UI(桌面)+Vant(移动) |
| v0.5 | 单端口部署（前端产物由后端托管，对外仅一个端口）；所有数据接口一律 POST，GET 只给页面/静态资源 |
| v0.6 | 后端使用 tsup 打包 |

最终确认的关键决策：① MVP 单用户 + 服务端多用户 + 多端同步 ② 预置 5 种类型、自定义作进阶(M4) ③ 图片上传服务端 uploads/ ④ SQLite 为服务端文件、经 REST API 同步 ⑤ 项目名 时光簿 / MomentLog ⑥ Naive UI(桌面) + Vant(移动 H5) ⑦ monorepo + pnpm workspace ⑧ 单级主题 ⑨ 标签默认任一匹配(可切全部) + 支持颜色 ⑩ 前后端共端口、前端产物由后端托管 ⑪ 全 POST 数据接口、GET 仅页面/静态 ⑫ 后端 tsup 打包、原生依赖 external。

## 2. 项目骨架

pnpm workspace monorepo：

```
.
├── package.json              # 根：workspace 入口，dev/build/start 脚本
├── pnpm-workspace.yaml       # workspace 包声明 + 原生构建白名单
├── pnpm-lock.yaml
├── .npmrc                    # 私域 npm 源（registry=http://10.8.0.10:4873/）
├── backend/                  # Fastify + TS + better-sqlite3，tsup 打包
└── frontend/                 # Vite + Vue3 + TS，构建产物 → backend/public
```

- **后端**：Fastify 单端口(:3000)托管前端 dist + uploads + 全部 POST 接口；建表 7 张（users / topics / events / records / record_types / topic_tags / record_tags）+ 种子（默认 user_id=1、5 种预置类型）；首次启动自动建库与 uploads 目录。
- **前端**：Vite + Vue3 + TS + Naive UI；4 页面（主题列表 / 事件时间线 / 事件详情 / 类型管理）+ 4 抽屉（事件 / 记录 / 主题 / 主题标签）；Axios 封装统一 POST。
- **接口**：全部 POST，动作子路径（/api/topics/list、/api/events/list、/api/topics/search 等）。

## 3. 构建与运行

```bash
pnpm install    # 安装全部依赖（根目录一次装齐）
pnpm dev        # 开发：前端 5173 + 后端 3000
pnpm build      # 生产构建：前端 → backend/public，后端 tsup → backend/dist
pnpm start      # 单端口运行后端（默认 http://localhost:3000）
```

环境变量：`PORT`（默认 3000）、`HOST`（默认 `0.0.0.0`，便于其他设备访问）。

## 4. 依赖安装问题与解决方案（重点）

### 4.1 fastify-static 包名修正
Fastify v4 的静态托管插件为 `@fastify/static`（非旧版 `fastify-static`）。已修正 `backend/package.json` 依赖与 `server.ts` 的 import。

### 4.2 私域 npm 源
初始 `pnpm install` 因默认 registry 在私域网络不可达而失败。已在项目根 `.npmrc` 配置私域源 `registry=http://10.8.0.10:4873/`，可正常拉取依赖。

### 4.3 pnpm supply-chain 策略拦截原生 build（核心问题）
- **现象**：`pnpm install` 在私域源配置正确、GitHub 可达的情况下仍报 `[ERR_PNPM_IGNORED_BUILDS] Ignored build scripts: better-sqlite3, esbuild, vue-demi`；被拦截 build 的包其**内部子依赖软链也被破坏**（`backend/node_modules/better-sqlite3` 能链上，但 `better-sqlite3` 内部的 `bindings` 子链接缺失、`.node` 原生文件未生成；`esbuild` 同理）。
- **根因**：pnpm 11 在本环境中的 supply-chain 安全策略（`Lockfile passes supply-chain policies`）强制忽略所有 build scripts，本地 `onlyBuiltDependencies` 配置不生效（警告 `package.json 的 pnpm 字段不再被读取`，且 workspace yaml 的 `onlyBuiltDependencies` 也被覆盖）。这是环境级限制，不是网络问题。
- **解决（当前设备已落地，可复用）**：
  1. `pnpm install --force --ignore-scripts` 强制重链全部依赖（跳过会卡死的 build）；
  2. 手动从 GitHub 下载 Node22 / win32-x64 预编译二进制 `better-sqlite3-v11.10.0-node-v127-win32-x64.tar.gz`，解压到 store 的 `better-sqlite3/build/Release/`（实测 1 秒可达，GitHub 通）；
  3. `bindings` 子链接在 `--force` 后已被 pnpm 自动重建；`esbuild` 的平台二进制 `esbuild.exe` 本就在 store，内部 `@esbuild/win32-x64` 软链也已被重建，无需手动干预；
  4. 验证：`node -e "require('better-sqlite3')"` 成功（sqlite 3.49.2）。
- **⚠️ 关键坑**：`pnpm install --force` 会清掉 store 里手动放的原生文件，因此若再次跑 force 安装，需重新执行步骤 2 把 `.node` 放回去（或统一用 `--ignore-scripts` + 手动原生文件流程）。
- **换设备提示**：原生模块需按目标平台重新生成。若目标环境可访问 GitHub 且 pnpm 未拦截 build，正常 `pnpm install` 会自动下载/编译对应平台原生模块；若被拦截或无法编译，按步骤 1–3 手动放置对应平台预编译二进制。

### 4.4 pnpm 的 deps status check 会中止 build/start
- **现象**：直接 `pnpm build` / `pnpm start` 报 `Command failed with exit code 1` 并提示 `Ignored build scripts`，根本没进入构建/启动。
- **原因**：pnpm 在运行 workspace 脚本前做依赖完整性检查，发现存在被忽略的 build scripts 即中止。
- **解决**：绕过 pnpm，直接调用底层工具（产物已可正常生成）：
  - 前端构建：`cd frontend && ./node_modules/.bin/vite build`（产物 → `backend/public`）
  - 后端构建：`cd backend && ./node_modules/.bin/tsup`（产物 → `backend/dist/server.js`）
  - 运行后端：`cd backend && node dist/server.js`（单端口 :3000）
  - 注：待环境允许 build scripts 后，可恢复 `pnpm build` / `pnpm start`。

### 4.5 前端 vite 构建报错（v-model on prop）
- **现象**：`vite build` 报 `[vue] v-model cannot be used on a prop`，位于 `TopicDrawer.vue` 等抽屉组件。
- **原因**：抽屉组件把父组件传入的 `show` prop 直接用于 `v-model:show="show"`，并在取消按钮里 `show = false` 给 prop 赋值，违反 Vue 单向数据流。
- **修复**：4 个抽屉组件（`TopicDrawer` / `EventDrawer` / `RecordDrawer` / `TopicTagDrawer`）统一改为：
  - `<n-drawer :show="show" @update:show="emit('update:show', $event)" ...>`
  - 取消按钮 `@click="emit('update:show', false)"`
  - 组件均已 `defineEmits(['update:show', 'saved'])`，`emit` 可用。

前次"后台安装中"的状态已闭环，当前项目**端到端可运行**：

- **接口验证**（curl / Node fetch）：
  - `POST /api/topics/list` → `{"code":0,"data":[],"message":""}`（统一响应格式正确）
