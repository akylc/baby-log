# 项目需求文档（PRD）

> 项目名称（暂定）：时光簿 / MomentLog
> 版本：v0.6（草稿）
> 更新时间：2026-07-11

---

## 1. 项目概述

### 1.1 背景
用户在生活或工作中经常需要在某个具体的时间点记录发生的事情，而且一件事往往包含多个维度的信息（文字、图片、链接、位置、标签等）。现有备忘录类工具大多只支持单一类型内容，难以把「同一时刻、不同角度」的信息聚合在一起。

### 1.2 目标
构建一个轻量的「时间点事件记录」应用：
- 用**主题（Topic）**把相关的事件归拢成一类合集（如「宝宝」「家庭」「旅行」），作为顶层管理单元；
- 以**时间点为锚**在某一主题下创建**事件（Event）**；
- 每个事件下可添加**多条记录（Record）**，每条记录对应**一种类型（Type）**；
- 通过**类型化记录**把同一时刻的不同信息结构化地保存与回看。

### 1.3 核心概念
| 概念 | 说明 |
|------|------|
| 主题 Topic | 顶层合集，把一类事件归拢在一起（如「宝宝」「家庭」「旅行」）。是事件的管理容器，含名称、图标、备注。 |
| 事件 Event | 某个主题下、一个时间点上发生的「事情」，是记录的容器。含标题与该时间点（occurred_at）。 |
| 记录 Record | 事件下的具体内容项，归属于某一种类型。 |
| 类型 Type | 记录的种类（如文本、图片、链接、位置、标签等），定义该记录的**内容结构**（决定内容怎么填）。 |
| 标签 Tag | 主题级的**分类标签**，用于对记录做横切标注与检索；每个主题拥有自己独立的一套标签，记录可勾选多个（非必选）。注意：与「标签类型」（FR-07）不同——前者是检索用的分类标签，后者是一种记录内容类型。 |
| 类型模板 | 系统预置 + 用户可自定义类型，决定记录的录入表单与展示样式。 |

> 关系：`主题 1—N 事件`，`主题 1—N 标签`，`事件 1—N 记录`，`类型 1—N 记录`，`记录 N—N 标签`。

## 2. 用户角色与使用场景

- **MVP 单用户（默认 user_id=1）**：自用记录，暂不实现登录；但**服务端按多用户架构设计**，数据统一存储在服务端（SQLite 文件位于服务端，不在前端），通过 REST API 实现多端同步，后期可平滑扩展多用户/多设备。
- **典型场景**：
  - 旅行中记录某天：文字感想 + 照片 + 打卡地点 + 标签；
  - 会议后记录某时刻：决议文本 + 附件链接；
  - 日常随手记：时间戳 + 一句话 + 标签。

## 3. 功能需求（FR）

### 3.0 主题（Topic）管理
- **FR-13** 创建主题：填写名称，可选图标、备注（如「宝宝」「家庭」「旅行」）。
- **FR-14** 查看主题列表（首页）：卡片/列表展示全部主题，显示每个主题下的事件数量；支持新建、进入。
- **FR-15** 编辑 / 删除主题：名称、图标、备注可改；删除主题**联动删除其下全部事件与记录**（需二次确认）。
- 事件必须归属某一主题（无「无主题」事件）。

### 3.1 事件管理
- **FR-01** 创建事件：选择所属**主题（Topic）**，填写标题，选择/自动带入发生时间（occurred_at），可加备注。
- **FR-02** 查看事件列表：在某个主题下（或跨主题「全部」视图）按发生时间倒序（或正序）展示，支持按日期/关键词筛选。
- **FR-03** 查看事件详情：展示该事件下全部记录，按类型分组或按时间排列。
- **FR-04** 编辑 / 删除事件：标题、时间、备注可改；删除事件联动删除其记录。

### 3.2 记录管理（类型化）
- **FR-05** 向事件添加记录：选择类型，按该类型模板填写内容；可从所属主题的标签集中**勾选标签**（可多选，非必选、可不选）。
- **FR-06** 编辑 / 删除记录（编辑时亦可增删标签）。
- **FR-07** 记录类型支持以下默认集合（可在 3.3 扩展）：
  1. 文本（text）：一段文字
  2. 图片（image）：上传到服务端专用文件夹，记录存储返回的可访问 URL
  3. 链接（link）：URL + 标题
  4. 位置（location）：地名 + 经纬度（可选）
  5. 标签（tag）：多个标签

### 3.3 类型与模板
- **FR-08** 查看可用类型列表。
- **FR-09**（进阶）用户自定义类型：定义类型 key、名称、图标、字段 schema。

### 3.4 检索与导出（进阶）
- **FR-10** 全局搜索：按关键词跨事件/记录检索。
- **FR-11** 按时间范围/类型过滤事件。
- **FR-12**（进阶）导出为 JSON / Markdown。

### 3.5 标签（Tag）管理
- **FR-16** 主题标签配置：每个主题可独立维护自己的标签集合（名称、可选颜色），支持新增 / 编辑 / 删除；标签随主题隔离，互不影响。
- **FR-17** 记录勾选标签：添加 / 编辑记录时，从当前主题标签集中以勾选形式选择（多选、非必选、可不选）。
- **FR-18** 主题内按标签检索：在某一主题下，可按一个或多个标签筛选记录（及其所属事件）；多选默认「任一匹配」，可切换「全部匹配」。
- **FR-19** 图片上传：上传图片至服务端专用文件夹（如 `uploads/`），返回可访问 URL；图片记录存储该 URL。MVP 由服务端本地目录管理，不做对象存储。

## 4. 数据模型（SQLite）

### 4.1 表结构
```sql
-- 用户表（多用户就绪；MVP 默认仅 user_id=1）
CREATE TABLE users (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  name       TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 主题表（顶层合集，归属某用户）
CREATE TABLE topics (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id    INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name       TEXT NOT NULL,
  icon       TEXT,                     -- 图标标识
  note       TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 主题标签表（每个主题独立一套）
CREATE TABLE topic_tags (
  id       INTEGER PRIMARY KEY AUTOINCREMENT,
  topic_id INTEGER NOT NULL REFERENCES topics(id) ON DELETE CASCADE,
  name     TEXT NOT NULL,
  color    TEXT,                      -- 展示色（可选）
  UNIQUE (topic_id, name)
);

-- 事件表
CREATE TABLE events (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  topic_id    INTEGER NOT NULL REFERENCES topics(id) ON DELETE CASCADE,
  title       TEXT NOT NULL,
  occurred_at TEXT NOT NULL,          -- ISO8601 时间点
  note        TEXT,
  created_at  TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 记录类型表（系统预置 + 可扩展）
CREATE TABLE record_types (
  id      INTEGER PRIMARY KEY AUTOINCREMENT,
  key     TEXT NOT NULL UNIQUE,        -- 如 'text','image'
  label   TEXT NOT NULL,               -- 显示名
  icon    TEXT,                        -- 图标标识
  schema  TEXT                         -- JSON: 字段定义
);

-- 记录表
CREATE TABLE records (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  event_id   INTEGER NOT NULL REFERENCES events(id) ON DELETE CASCADE,
  type_id    INTEGER NOT NULL REFERENCES record_types(id),
  payload    TEXT NOT NULL,            -- JSON: 具体字段值
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 记录-标签关联表（多对多）
CREATE TABLE record_tags (
  record_id INTEGER NOT NULL REFERENCES records(id) ON DELETE CASCADE,
  tag_id    INTEGER NOT NULL REFERENCES topic_tags(id) ON DELETE CASCADE,
  PRIMARY KEY (record_id, tag_id)
);

CREATE INDEX idx_events_topic ON events(topic_id);
CREATE INDEX idx_records_event ON records(event_id);
CREATE INDEX idx_record_tags_record ON record_tags(record_id);
CREATE INDEX idx_record_tags_tag    ON record_tags(tag_id);
```

### 4.2 字段示例
- `topics`：`{"name":"宝宝","icon":"baby","note":"记录宝宝成长点滴"}`
- `topic_tags`：`{"name":"疫苗","color":"#f56c6c"}`（归属某 topic_id）
- `record_tags`：关联记录，如 `{ "record_id": 12, "tag_id": 3 }`
- `record_types.schema`（JSON）：`{"fields":[{"name":"content","type":"string","label":"内容","required":true}]}`
- `records.payload`（JSON）：`{"content":"今天天气真好"}`
- 图片文件：存于服务端 `uploads/` 目录，记录 payload 存返回 URL，如 `{"url":"/uploads/2026/abc.jpg"}`

## 5. 技术方案

### 5.1 整体架构
```
┌───────────────────────── 单端口（如 :3000） ─────────────────────────┐
│  ┌─────────────┐   POST(数据) / GET(页面·资源)    ┌──────────┐       │
│  │  Vue3 SPA   │  ◄──────────────────────────►  │ SQLite3  │       │
│  │ (Vite+TS)   │   Fastify 同端口托管前端+接口   │ (文件库) │       │
│  └─────────────┘                                └──────────┘       │
└─────────────────────────────────────────────────────────────────────┘
```
- 前端为纯静态 SPA，**构建产物（dist/）由 Fastify 统一托管**；前端与后端**共用同一个端口**（如 `:3000`），对外只暴露这一个端口，无需独立部署前端或额外反向代理；
- 后端**既提供数据接口、又托管前端页面与静态资源**：数据请求统一使用 `POST`，页面与静态资源使用 `GET`（Fastify 静态托管 + SPA history 回退到 `index.html`）；所有读写经服务端，服务端为唯一数据源；
- SQLite 单文件数据库位于**服务端**（非前端本地），作为服务端持久化文件，零运维、易备份；多端同步经由 REST API 实现。

### 5.2 前端（Vite + Vue3 + TS）
- **包管理器**：统一使用 **pnpm**（项目所有前端依赖安装与脚本运行均通过 `pnpm`，不使用 npm/yarn）
- **构建**：Vite
- **框架**：Vue 3（`<script setup>` + Composition API）
- **语言**：TypeScript（严格模式）
- **路由**：Vue Router（主题列表 / 事件列表 / 事件详情 / 类型管理）
- **状态**：Pinia
- **响应式 / 移动端**：采用移动优先（Mobile-First）设计，弹性布局 + 媒体查询 / 组件库栅格，桌面与手机自适应；核心页面在窄屏（≤768px）单列展示、操作按钮适配触控。
- **UI 组件库**（候选，请选择其一，对比见下表）：
  - **Naive UI**：轻量、TS 友好、主题定制灵活，Vue3 原生支持好（桌面端推荐）
  - **Element Plus**：生态成熟、组件最全、中文文档完善（桌面端）
  - **Vant**：若重度面向移动端，Vant 是 Vue3 移动端组件库首选（轻量、触控友好）
  - **Varlet**：Vue3 移动端组件库，Material 风格，体积小（移动端）
- **请求**：统一使用 **Axios** 封装（拦截器、统一错误处理、cancel token），类型与后端共享 `types` 定义；**所有数据接口统一 `POST`，查询/筛选参数均放在请求体**，不使用 GET 传数据。

| 组件库 | 定位 | TS 支持 | 体积 | 适用端 | 备注 |
|--------|------|---------|------|--------|------|
| Naive UI | 通用桌面 | 优 | 中 | 桌面 | 主题灵活，Vue3 原生 |
| Element Plus | 通用桌面 | 优 | 偏大 | 桌面 | 组件最全，文档好 |
| Vant | 移动端 | 优 | 小 | 移动/H5 | 触控体验好，电商常用 |
| Varlet | 移动端 | 优 | 小 | 移动/H5 | Material 风格 |

> 选型建议：若桌面 + 移动都要，可 Naive UI（桌面）+ Vant（移动 H5）按需组合，或优先保证移动端用 Vant/Varlet；纯桌面用 Naive UI / Element Plus。
- **目录结构（建议）**：
```
src/
  api/         # 接口封装
  components/  # 通用组件（记录卡片、类型选择器等）
  views/       # 页面
  stores/      # Pinia
  types/       # 共享 TS 类型
  utils/
```

### 5.3 后端（Fastify + TS + SQLite3）
- **包管理器**：统一使用 **pnpm**（后端依赖安装与脚本运行同样使用 `pnpm`，与前端保持一致）
- **框架**：Fastify（高性能、插件化、内置校验）
- **语言**：TypeScript（源码由 **tsup** 打包为可运行产物，默认输出 ESM/CJS 到 `dist/`）
- **数据库**：`better-sqlite3`（同步 API，简单可靠）
- **校验**：Fastify 的 schema（基于 JSON Schema）做请求校验
- **构建/打包**：使用 **tsup** 将后端 TS 源码打包为精简产物（默认输出 `dist/server.js` 等单文件入口）；原生模块（如 `better-sqlite3`）设为 `external` 不参与打包；通过 `pnpm build`（调用 `tsup`）触发，产物可直接 `node dist/server.js` 运行。
- **关键依赖**：`fastify`, `fastify-static`(托管前端 dist 与 `uploads/`、含 SPA history 回退), `@fastify/multipart`(图片上传), `better-sqlite3`, `zod`(可选，配合 schema)
- **单端口说明**：前端构建产物部署到后端的 `public/`(或 `dist/`) 目录、由 `fastify-static` 托管；因前端与接口**同源同端口**，**默认无需 `fastify-cors`**（若后续前端独立部署再启用）。所有数据接口均为 `POST`。
- **目录结构（建议）**：
```
src/
  server.ts       # 启动入口
  plugins/        # 数据库、cors 等插件
  routes/         # /topics, /events, /records, /types
  db/             # 建表、迁移、查询封装
  types/          # 共享类型
  schemas/        # 请求/响应 JSON Schema
```

## 6. 接口设计（全 POST 数据接口 · 同源单端口）

| 方法 | 路径 | 说明（请求体关键字段） |
|------|------|------|
| POST | `/api/topics/list`            | 主题列表（含各主题事件数） |
| POST | `/api/topics/create`          | 创建主题 |
| POST | `/api/topics/detail`          | 主题详情（`{id}`） |
| POST | `/api/topics/update`          | 编辑主题（`{id, ...}`） |
| POST | `/api/topics/delete`          | 删除主题（`{id}`，联动删事件与记录） |
| POST | `/api/topics/tags`            | 某主题标签集合（`{topicId}`） |
| POST | `/api/topics/tags/create`     | 新增主题标签 |
| POST | `/api/topics/tags/update`     | 编辑主题标签（`{tagId, ...}`） |
| POST | `/api/topics/tags/delete`     | 删除主题标签（`{tagId}`） |
| POST | `/api/topics/search`          | 主题内检索（`{topicId, tags:[], match:'any'\|'all', q}`） |
| POST | `/api/events/list`            | 事件列表（`{topicId?, from?, to?, q?, tags?}`，不传 topicId 表示全部） |
| POST | `/api/events/create`          | 创建事件 |
| POST | `/api/events/detail`          | 事件详情（`{id}`，含记录） |
| POST | `/api/events/update`          | 编辑事件 |
| POST | `/api/events/delete`          | 删除事件 |
| POST | `/api/events/records`         | 某事件下记录列表（`{eventId}`） |
| POST | `/api/events/records/create`  | 添加记录（可含 `tags: number[]`） |
| POST | `/api/records/update`         | 编辑记录（可同步更新 `tags`） |
| POST | `/api/records/delete`         | 删除记录 |
| POST | `/api/types/list`             | 记录类型列表 |
| POST | `/api/types/create`           | （进阶）新增自定义类型 |
| POST | `/api/upload`                 | 上传图片到服务端 `uploads/`，返回 URL |

> **约定**：`GET` 仅用于前端页面（Vue Router history 模式回退到 `index.html`）与静态资源（JS/CSS/图片等由 `fastify-static` 托管）；**所有数据读写接口一律使用 `POST`**，查询/筛选条件放在请求体中。
> 统一响应：`{ "code": 0, "data": ..., "message": "" }`；错误用 HTTP 状态码 + message。
> 多用户说明：服务端按多用户建模，核心数据按 `user_id` 隔离；MVP 不实现鉴权，所有请求默认归属 `user_id=1`，预留后续登录/令牌扩展。

## 7. 页面与交互（MVP）

1. **主题列表页（首页）**：卡片/列表展示全部主题（如「宝宝」「家庭」「旅行」），每张卡显示主题名、图标、事件数量；提供「新建主题」；点击进入该主题的事件时间线；可进入「标签管理」。
2. **事件时间线页**：在某一主题下（或从「全部」视图）展示卡片流，按发生时间排列；顶部增加**标签筛选区**（展示该主题标签，可勾选，单选/多选）+ 搜索栏；「+ 新建事件」需选定所属主题。
3. **事件详情页**：标题 + 时间 + 备注；下方按类型展示记录卡片（标注已选标签）；「添加记录」按钮。
4. **添加/编辑记录抽屉**：选择类型 → 渲染对应表单（由 type schema 驱动）；下方增加**标签勾选区**，从所属主题标签集中多选（非必选、可不选）。
5. **添加/编辑主题抽屉**：填写名称、图标、备注（从主题列表页唤起）。
6. **主题标签管理抽屉**：在当前主题内增 / 改 / 删标签（名称、颜色），配置该主题**独立**的标签集。
7. **类型管理页**（进阶）：查看/新增自定义类型。

## 8. 非功能需求
- **NFR-01** 服务端存储 + 多端同步：数据统一存于服务端 SQLite（非前端本地），通过 REST API 供多端访问与同步；MVP 默认单用户（user_id=1，无登录），但服务端按多用户建模，预留鉴权扩展点。
- **NFR-02** 启动简单：开发期统一使用 pnpm，`pnpm install && pnpm run dev` 同时起前后端（monorepo / pnpm workspace）；生产期仅运行后端，由后端托管前端构建产物。
- **NFR-03** 类型安全：前后端共享 TS 类型定义。
- **NFR-04** 性能：事件列表、记录渲染在千级数据下无卡顿。
- **NFR-05** 数据可备份：SQLite 单文件可直接复制迁移。
- **NFR-06** 响应式：桌面与移动端自适应，核心交互在窄屏下可用、触控友好。
- **NFR-07** 多用户就绪：核心表带 `user_id`，查询按用户隔离；MVP 不实现账号体系，所有请求默认归属 user_id=1，预留鉴权扩展点。
- **NFR-08** 单端口部署：前后端共用同一端口（如 `:3000`），前端构建产物由后端托管，对外仅暴露一个端口，无需额外反向代理。
- **NFR-09** 接口方法约定：所有数据接口统一 `POST`；`GET` 仅用于前端页面与静态资源。

2. **类型集合（已确认）**：采用预置 5 种类型（文本/图片/链接/位置/标签）；用户自定义类型作为进阶（M4）实现，MVP 不做。
3. **图片存储（已确认）**：图片上传到**服务端本地专用文件夹**（如 `uploads/`）统一管理，记录存返回的可访问 URL；MVP 不做对象存储。
4. **多端同步（已确认）**：需要。SQLite 是**服务端**存储文件（非前端本地），多端同步由服务端 REST API 实现；客户端不直连数据库。

6. **UI 组件库（已确认）**：桌面端 **Naive UI** + 移动 H5 **Vant** 组合（按需引入）。
7. **工程组织（已确认）**：前后端同仓 monorepo，根目录 `pnpm-workspace.yaml`，下分 `frontend/`、`backend/`；包管理器一律 pnpm。
8. **主题层（已确认）**：「主题(Topic)」为顶层容器，首页为主题列表页；事件必须归属某一主题（无「无主题」事件）；MVP 采用**单级主题**，不做多级分组。
