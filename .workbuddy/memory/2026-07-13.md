## 修复：跨天首页日期不更新的 bug（重要）
- 现象：23:59 进入应用，00:01 记录后返回首页，新一天的记录不显示、怎么刷新都卡在前一天。
- 根因：`Home.vue` 第 304 行 `const today = todayStr()` 是组件**首次挂载时**固定的字符串常量；首页用 `<keep-alive>`，跨天后 `today` 永不过期。`refresh()` 查询窗口 `[今天-6, 今天]` 用旧 today → 新日期记录落在窗口外没被拉取；且「后一天」按钮被 `:disabled="selectedDate === today"` 禁用，翻不过去。
- 修复：
  - `today` 改为 `ref(todayStr())`，所有脚本引用补 `.value`（模板第 28 行 ref 自动拆包无需改）。
  - `refresh()` 开头重新取真实 `newToday`；若 `selectedDate === 旧today` 且跨天，置 `internalDateChange=true` 后更新 `selectedDate`/`selectedTs` 跟随到新一天；`today.value = newToday`。
  - `watch(selectedTs)` 加 `internalDateChange` 守卫，避免程序化改日期触发二次刷新（防递归/重复请求）。
  - 历史日期视图（selectedDate 非今天）不受影响，`today` 仍会更新使「今日/昨天」标签正确。
- 验证：`pnpm --filter frontend build` 通过；已热更新 `backend/dist/public`（rm assets + cp），:26712 返回 200。

## 修复：iOS 书签模式（display-mode:standalone）首页 phantom 滚动条 + 底部留空过大
- 现象：iOS「添加到主屏幕」独立模式下，记录未超一屏仍出现滚动条，且底部留空过大。
- 根因（两处叠加）：
  1. `100dvh` 在 standalone 模式下取整有 1px 级误差，使 `overflow-y:auto` 的 `.page-area` 误判内容超屏而冒出滚动条（phantom scrollbar）。
  2. 底部安全区被重复预留：`.app-shell` 已 `padding-bottom: env(safe-area-inset-bottom)`（独立模式 34px），而 `Home.vue .build-info` 又加了 `calc(20px + env(safe-area-inset-bottom))`——两份叠加既留空过大，又让滚动高度被多算。
- 修复：
  - `App.vue .page-area` 加 `overscroll-behavior-y:contain`，抑制橡皮筋把滚动条顶出。
  - `Home.vue .build-info` 的 `padding: 18px 0 calc(20px + env(safe-area-inset-bottom))` → `padding:18px 0 20px`，去掉重复底部安全区（统一由 `.app-shell` 预留，所有页面通用）。FAB 右下角、build-info 居中，横向不重叠，无冲突。
- 验证：`pnpm --filter frontend build` 通过；已热更新，:26712=200；grep 确认产物 CSS 含 `overscroll-behavior-y:contain`、`.build-info{...padding:18px 0 20px}`。
- **纠正（同次）**：曾一度加 `@media (display-mode:standalone){ .app-shell{height:100%} }` 想用精确高度消除 dvh 取整的 phantom 条，但 iOS 独立 WebView 里 `100%` 高度链未解析成确定高度，导致 `.page-area` 失去约束、`overflow-y:auto` 拿不到可滚动区域（外层 overflow:hidden 把超出裁掉）→ **书签模式完全不能滚动**。已撤销该覆盖，恢复 `100dvh`（原先本就能滚动）。phantom 条根因实为底部安全区被重复预留（已靠去掉 build-info 的 safe-area 省出 34px 余量，足以覆盖 dvh 取整的 1px 误差，phantom 自然消失）。

## Dockerfile：给 VERSION 加「缺失即失败」守卫（修复 version 标签为空）
- 用户反馈「docker 的别名 version 空了」。根因：Dockerfile 原 `ARG VERSION`（无默认值）+ `LABEL version=$VERSION`。一旦 `docker build` 漏传 `--build-arg VERSION=...`，`$VERSION` 为空 → 镜像 `version` LABEL 为空（即 Docker 界面里看到的「别名/版本」空）。

## 修复：版本握手误报「当前版本过低」（前端 0.0.10 vs 后端 0.0.9 矛盾提示）
- 用户反馈：页面提示「当前版本 v0.0.10，最新版 v0.0.9」的荒谬弹框。
- **根因（两层）**：① 部署状态错配——运行中的后端是手动 node 进程（旧 PID 11040），内存里 `APP_VERSION=0.0.9`（其加载的是旧 dist/server.js；而磁盘 dist 已被多次 build 覆盖成 0.0.10、前端静态产物也已 0.0.10）。前端携 `X-App-Version:0.0.10` 请求，后端比的是 0.0.9。② 握手逻辑缺陷——原判断 `cmpVersion(client, server) !== 0` 把**任何不一致**都当「前端过低」弹框；当**前端比后端新**（发版后前端先生效、后端进程未重启的窗口期）也弹框，且 App.vue 弹框标题写死「当前版本过低」，于是出现"我都 0.0.10 了怎么 0.0.9 才是最新"的矛盾文案。
- **修复（server.ts）**：把判定从 `!== 0` 改为 `cmpVersion(client, server) < 0`——**仅当「前端版本 < 后端版本」才拒绝弹框**；前端版本 >= 后端一律放行。前端更新时不再误报；只有真停留在旧前端（前端 < 后端）才提示刷新。删除了原 `low` 分支的 `message`（现在只会出现前端旧这一条路径）。
- 顺带根治部署错配：重启后端进程（kill 11040，从 backend/ 重启新 dist），使运行实例 `APP_VERSION` 也对齐到 0.0.10，两端完全一致。

- 教训：发版时必须**后端进程也重启**到新 dist，否则「前端新/后端旧」窗口期 + 旧逻辑会持续误弹框。新逻辑已让该窗口期不再误报，但 health 版本仍应以重启后的后端为准。
