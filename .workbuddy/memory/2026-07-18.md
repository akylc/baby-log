# 2026-07-18 工作日记

- **扇形菜单排序不一致的根因与修复（重要）**：用户反馈开发环境「首页扇形菜单」与「添加记录页扇形菜单」排序仍不一样。排查发现根因不在数据源，而在**加载逻辑**：
  - 首页扇形 = `Home.vue` 的 `<PieTypeMenu :items="FILTER_OPTIONS">`，`FILTER_OPTIONS` 直接用代码固定顺序 `RECORD_TYPES`，**从不读 localStorage**。
  - 添加记录页扇形 = `Record.vue` 的 `<PieTypeMenu :items="orderedTypes">`，`orderedTypes` 来自 `loadTypeOrder()` 读取的 **localStorage `ml_type_order`**——这是用户在添加页拖拽芯片重排过的**自定义顺序**，持久化在浏览器。
  - 因此开发环境（用户浏览器已存自定义顺序）两边不一致；之前「统一数据源」只对齐了默认顺序，未让两个扇形共用同一份 localStorage 顺序。
  - **修复**：把 `loadTypeOrder/saveTypeOrder/orderTypeOrder/TYPE_ORDER_KEY` 抽到共享模块 `frontend/src/constants/recordTypes.ts`；`Home.vue` 的 `FILTER_OPTIONS` 改为 `computed(() => orderedRecordTypes(loadTypeOrder()))`，让首页扇形也读 `ml_type_order`；`Record.vue` 删除本地重复定义、改用共享函数、`order` 类型改 `RecType[]`、调用点 `saveTypeOrder(order.value)`。现在无论用户是否拖拽重排，两个扇形永远一致。
  - 注：若用户想让两处都回到默认顺序，需清除浏览器 `ml_type_order`（或后续加「重置排序」入口）。
- 文件改动：`frontend/src/constants/recordTypes.ts`（新增共享顺序读写）、`frontend/src/views/Home.vue`、`frontend/src/views/Record.vue`（源码未提交，待用户「提交/发版」指令）。`pnpm --filter frontend build` 通过。
